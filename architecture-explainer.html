TH Address Latent / System explainer

แยกที่อยู่ไทย
ด้วย กติกา + ข้อมูลพื้นที่ + คะแนน

เอกสารนี้อธิบายระบบสำหรับคนที่เคยทำ frontend มาก่อน: ไม่ต้องรู้ Machine Learning ก็เห็นได้ว่าข้อความหนึ่งบรรทัดเดินทางผ่านระบบอย่างไรจนกลายเป็นข้อมูลที่ใช้ใน UI ได้

แก่นของระบบ
มันไม่ได้ทายคำตอบจาก LLM แต่สร้างตัวเลือกหลายแบบ แล้วเลือกชุดที่เข้ากันที่สุด พร้อมยอมเว้นช่องที่ไม่มั่นใจ
ภาพรวมPipelineLatent คืออะไรทำไมต้องตรวจซ้ำแผนที่ไฟล์สถานะจริง

02 / pipeline

การทำงานจริงมี 4 ด่าน

ให้คิดเหมือน form parser ที่ไม่เชื่อคำตอบแรกของตัวเอง: หาให้กว้าง เลือกอย่างมีบริบท แล้วตรวจอย่างเข้มก่อนส่งออก

01

เตรียม resource

ใช้ dataset สร้างเวกเตอร์ และโหลด gazetteer ที่บอกความสัมพันธ์ของตำบล อำเภอ จังหวัด และไปรษณีย์

02

สร้าง candidate

เจอเบอร์โทร จังหวัด หรือช่วงข้อความที่อาจเป็นชื่อ/ที่อยู่ แล้วเก็บเป็นตัวเลือกพร้อมคะแนน

03

เลือกชุดคำตอบ

ตัดตัวเลือกที่ทับกัน หรือจับคู่ location ไม่เข้ากัน ด้วย beam search

04

ตรวจผล

normalize format, ตรวจ offset และยอมคืน null เมื่อข้อมูลขัดแย้งหรือมั่นใจต่ำ

Candidate generator
→
Regex หา phone/postcode, lookup จาก gazetteer และ heuristics แบ่งช่วงชื่อหรือรายละเอียดที่อยู่
Latent scoring
→
ให้คะแนนว่า candidate แต่ละช่วง “หน้าตาคล้าย” label นั้นเพียงใด
Pruner
→
เลือกชุด candidate ที่คะแนนรวมดีและไม่ชนกัน แทนการเลือก field แบบอิสระ
Validator
→
ตรวจ format และความสอดคล้องของพื้นที่ก่อนคืน ParseResult

03 / latent, without magic

“Latent” ในโปรเจกต์นี้ไม่ใช่ GPT

ภาษาคน: ระบบเรียนรู้รูปทรงของข้อความแต่ละประเภทไว้เป็นไม้บรรทัดหลายอัน แล้วเอาข้อความใหม่มาเทียบว่าคล้ายไม้บรรทัดของ NAME, PHONE หรือ ADDRESS_DETAIL แค่ไหน
ช่วงข้อความ

เช่น นายทดสอบ ใจดี
มอง n-gram, ความยาว, มีเลขไหม, มีอักษรไทยไหม และบริบทซ้าย/ขวา

→
Feature vector

แปลงลักษณะเป็นตัวเลขขนาดคงที่ เช่น 256 ช่อง แล้วทำ normalization

→
Direction ของ label

เปรียบเทียบกับเวกเตอร์ที่สรุปจากตัวอย่าง label นั้น แล้วแปลงเป็นคะแนน 0–1

NAME 0.81

สิ่งที่ “มี”

character n-grams
feature hashing
dot product
sigmoid score

เป็น linear scoring ที่ deterministic: input และ resource เดิมจะได้คำตอบเดิม

สิ่งที่ “ไม่มีตอนรัน”

no LLM
no API call
no backpropagation
no random sampling

browser ไม่ได้ train และไม่ส่งข้อความผู้ใช้ไป server

คะแนนสุดท้าย

0.55 × latent
+ 0.45 × evidence

หลักฐานแบบ regex หรือ gazetteer ยังมีน้ำหนักมาก จึงเป็น hybrid parser

แล้ว direction ถูกสร้างขึ้นมาอย่างไร?

ตอน build resource ระบบเอา feature เฉลี่ยของตัวอย่าง label หนึ่ง ลบด้วย feature เฉลี่ยของ label อื่น แล้ว normalize ผลลัพธ์เป็น direction ของ label นั้น ดู src/latent/fit.ts

04 / choosing safely

ทำไมเลือก candidate ที่คะแนนสูงสุดทีละช่องไม่ได้

ปัญหา: ข้อความหนึ่งช่วงตีความได้หลายแบบ

ปทุมวัน อาจเป็นทั้งตำบลและอำเภอ และข้อมูลแต่ละ field อาจขัดแย้งกันได้ถ้าเลือกแยกคนละรอบ

Pruner จึงเลือกเป็น “ชุด” โดยห้ามช่วงข้อความทับกัน และให้โบนัสเมื่อ tuple พื้นที่เข้ากัน

ตัวอย่าง: postcode ขัดกับพื้นที่

ถ้าข้อความมี ปทุมวัน กรุงเทพมหานคร 99999 ระบบอาจอ่านรูปแบบ 5 หลักได้ แต่ validator เห็นว่าไม่สอดคล้องกับ gazetteer

ผลลัพธ์ที่ถูกต้องกว่าคือไม่ใส่ postcode

ACCEPT

ข้อมูลมั่นใจและเข้ากัน

คืนค่า canonical เช่นลบขีดจากเบอร์โทร และเก็บตำแหน่งของข้อความไว้ใน spans

ABSTAIN

ไม่แน่ใจหรือข้อมูลขัดกัน

คืน null พร้อม reason เช่น low-confidence, invalid-format, inconsistent-location

05 / code map

เปิดอ่านโค้ดตามลำดับนี้จะเข้าใจเร็วที่สุด

src/App.tsxReact demo: รับ input, เรียก parser, แสดงผล
src/parser.tsจุดรวม pipeline และ public API
src/candidates.tsหา candidate และรวม evidence score กับ latent score
src/decode/pruner.tsbeam search เลือกชุด candidate ที่เข้ากัน
src/validate/result.tsตรวจ format, location consistency และสร้างผลลัพธ์
src/latent/features.tsข้อความ → feature vector → คะแนน direction
src/latent/fit.tsสร้าง frozen direction จาก labeled dataset
scripts/build-resources.tsสร้าง JSON resource ที่ browser import ไปใช้

06 / current verdict

ตอนนี้ latent ยังเป็นส่วนทดลอง

Benchmark ปัจจุบันใช้ข้อมูล synthetic และวัด exact-record accuracy โดยเปรียบเทียบ pipeline เดิมที่เปิด/ปิด frozen direction

10.1%Hybrid + latent directions
12.7%Hybrid ไม่มี directions
1.3%Legacy regex parser

ดังนั้นสิ่งที่ใช้งานได้ชัดคือโครง parser แบบ hybrid, gazetteer และ validation ส่วน latent direction ยังต้องพิสูจน์ว่าช่วยได้จริงก่อนใช้เป็นค่า default
