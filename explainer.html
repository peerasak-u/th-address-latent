Thai Address latent ไม่ได้พยายามทายที่อยู่ทั้งก้อนในครั้งเดียว แต่มันสร้างผู้สมัครเล็ก ๆ จากข้อความ แล้วใช้ frozen character n-gram directions, คะแนนหลักฐาน และกฎความสอดคล้องของสถานที่ ค่อย ๆ เลือกคำตอบที่มีเหตุผลที่สุด

คำถามที่หน้านี้จะตอบทำไมระบบใหม่มี if น้อยลง แต่ยังรักษาความเบาและทำงานใน browser ได้? คำตอบอยู่ที่การย้ายความซับซ้อนจากกฎรายคำ ไปเป็น representation + search + validation

01 · The shift

Legacy กับ latent คิดคนละแบบ

ของเดิมเป็น pipeline แบบทำความสะอาด → ตัดคำ → match → ลบคำ location → เอาที่เหลือเป็น address ซึ่งตรงไปตรงมาและอ่านง่าย แต่ทุก edge case ใหม่มักกลายเป็น regex หรือ special case ใหม่

thai-address-splitter

แนวคิดหลักคือรู้จัก pattern ให้มากพอ: prefix, regex, คำ location, และลำดับการลบข้อมูล

clean(text)
extractThaiWords(text)
matchLocations(words)
remove(locationWords)
extractNamePhone(remaining)
remaining → address

th-address-latent

ไม่ต้องบอกทุกคำว่าอยู่ field ไหน แต่ให้หลายคำตอบที่เป็นไปได้ แล้วให้ scorer/pruner/validator ร่วมกันตัดสิน

generateCandidates(raw)
score(evidence, frozenDirection)
prune(no-overlap, location-coherence)
validate(or abstain)

04 · Character n-grams
n-gram คือกล้องหลายระยะ
แทนที่จะต้อง tokenize ภาษาไทยให้ถูกก่อน ระบบมองข้อความเป็นหน้าต่างตัวอักษรหลายขนาด จึงจับคำติดกัน, คำย่อ, punctuation และ typo บางส่วนได้โดยไม่ต้องมี dictionary rule ทุกแบบ

        

          

          
feature = hash("gN:" + window)
ทุก window ถูก hash ลง vector ขนาด 512 แล้ว normalize เป็นจุด representation เดียวกัน ทำให้ runtime ใช้แค่ typed array + dot product ไม่มี model server และไม่มี training ตอนใช้งาน
สกลนคร → `ส`, `กล`, `นคร` ... → vector

The mental model
จากกฎรายคำ → การรวมหลักฐาน
ระบบใหม่ไม่ได้ลบความรู้เชิงกฎทิ้ง แต่ย้ายมันไปอยู่คนละชั้น: candidate generation ให้โครงสร้าง, latent ให้ soft signal, gazetteer ให้ identity ของ location, pruner ให้ความสัมพันธ์, validator ให้สิทธิ์ในการ abstain
  • ไม่มี magic model ที่เดาคำตอบทั้งบรรทัด
  • ไม่มี runtime training หรือ API
  • ถ้า evidence ไม่พอ ระบบเลือกไม่ตอบได้
  • ทุกชั้น inspect และ benchmark แยกกันได้
